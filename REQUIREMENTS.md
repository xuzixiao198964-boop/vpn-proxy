# VPN 代理转发项目 — 需求说明

> 本文档描述产品能力与约束，不涉及具体编程语言、框架与开发环境配置。

---

## 1. 项目背景与目标

在自有 VPS（搬瓦工等）上部署**服务端转发能力**，在 **Windows** 与 **Android** 上各提供一款**本地代理客户端**。用户设备上的应用流量经客户端加密后送至服务端，由服务端进行**出站转发**，实现统一出口与访问策略。  
（是否采用标准 VPN 隧道形态或应用层代理形态，由后续技术方案确定；本文档从业务需求侧描述行为与体验。）

---

## 2. 角色与使用场景

| 角色 | 说明 |
|------|------|
| 终端用户 | 在 Windows/Android 上安装客户端，登录后启用代理，使指定应用或系统流量经服务端转发 |
| 管理员 | 维护服务端、用户账号、转发策略与运行状态 |

**典型场景**：办公网络受限、需要固定出口 IP、多设备统一代理策略等。

---

## 3. 功能需求

### 3.1 账号与认证（必须）

- 支持**用户名 + 密码**登录；密码在传输与存储侧需满足安全基线（如传输加密、服务端不可逆存储）。
- 支持**登出**；会话失效后需重新登录方可使用代理。
- 登录失败需有明确提示（不泄露账号是否存在等敏感信息，可统一为「用户名或密码错误」）。
- （可选）后续可扩展：修改密码、找回密码、多设备会话管理。

### 3.2 Windows 客户端

- 安装或解压后即可配置**服务端地址**（域名或 IP + 服务端口，与 SSH 管理端口区分）。
- 登录成功后，可**启动/停止**本地代理服务。
- 提供**系统代理**或**指定应用走代理**的能力（具体范围在实现阶段细化，需求上需支持「用户可感知开关与基本状态」）。
- 异常时给出可读错误信息（网络不可达、认证失败、服务端维护等）。

### 3.3 Android 客户端

- 支持配置服务端地址并完成登录。
- 登录后可启用本地代理；在 Android 上可通过 **VPN 服务接口** 或 **本地代理 + 应用分流** 等方式实现（实现阶段选型），需求上要求：**用户可一键开关**，并显示**连接/未连接**等状态。
- 需考虑省电与后台策略：断线重连、网络切换提示等（基线可先做简单重试与状态提示）。

### 3.4 服务端（VPS 转发）

- 接收已认证客户端的连接，**仅允许合法用户**使用转发能力。
- 将客户端流量按策略**转发至公网**（出站），并可记录**必要审计信息**（时间、用户标识、流量概要等，符合合规前提下最小化采集）。
- 支持**基本运维**：进程可用性、配置热更新或重启后配置持久化、日志轮转。
- **SSH（22）** 仅用于运维管理，与业务代理端口分离；业务端口在防火墙中按需开放。

### 3.5 安全与合规

- 客户端与服务端之间通信**必须加密**，防窃听与中间人攻击（证书校验等基线）。
- 不在客户端持久化明文密码；会话令牌安全存储与过期策略需明确。
- 用户需知悉：本系统用于合法网络访问，遵守当地法律与服务商条款。

---

## 4. 非功能需求

- **可用性**：核心路径（登录 → 开代理 → 访问外网）在常见网络下可用；断网有明确反馈。
- **性能**：满足日常浏览与常见应用延迟可接受（具体指标可在测试阶段量化）。
- **可维护性**：配置项（服务端地址、端口等）对用户可见且可修改；日志便于排查问题。

---

## 5. 范围外（本期可不实现）

- iOS / macOS / Linux 客户端（可列为后续阶段）。
- 复杂的多租户计费与套餐系统（除非单独立项）。
- 违法用途相关能力。

---

## 6. 验收要点（摘要）

- Windows 与 Android 均能使用**同一套账号**登录服务端并完成代理转发。
- 未登录或会话无效时，**不能使用**转发能力。
- 服务端可独立部署在指定 VPS，且业务端口与 SSH 管理职责清晰分离。
